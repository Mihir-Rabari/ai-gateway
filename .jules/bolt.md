## 2025-04-22 - Atomic Redis Operations for Rate Limiting
**Learning:** Sequential Redis operations (`incr` followed by `expire` in an `if` block) are susceptible to race conditions under high concurrency. If the application crashes between the operations, or if multiple concurrent requests increment the key simultaneously before it expires, the key might never receive an expiration time (TTL), resulting in an infinite lock or rate limit leak.
**Action:** Always use atomic Lua scripts via `redis.eval()` for operations that require modifying a key and simultaneously setting its TTL to guarantee correctness without requiring full distributed locking.
